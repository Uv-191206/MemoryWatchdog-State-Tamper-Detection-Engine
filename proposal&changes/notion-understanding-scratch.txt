1. The Background Concept: Virtual Memory & Process Isolation
Before reading the API documentation, you must understand the problem WinAPI is solving: Process Isolation.

Modern operating systems like Windows do not let programs share the same physical RAM directly. Instead, the OS assigns every program its own private "Virtual Memory" space. If your target application (the shipwreck game) stores collectedWood = 10 at a memory address, your hack/watchdog program cannot simply read that address, because it exists in an entirely different virtual universe.

To cross this boundary, your C++ programs must politely ask the Windows Kernel for permission. This is where the <windows.h> library comes in.

2. The <windows.h> Arsenal (Your Reading List)
When looking up documentation on Microsoft Learn or searching for technical papers, you should focus on the Memory API (memoryapi.h) and the Tool Help Library (tlhelp32.h), which are both included via <windows.h>.

Here are the specific functions to research, along with their roles and background mechanics:

A. Finding the Target: CreateToolhelp32Snapshot
The Role: Neither your Watchdog nor your Hack inherently knows that your target application is running. This function acts like a radar sweep.

How it works in the background: When you call this, WinAPI asks the OS to freeze a momentary "snapshot" of all running processes, threads, and modules currently active in the system. You then use related functions like Process32First and Process32Next to iterate through this list until you find the exact Process ID (PID) of your target .exe.

Documentation Search Term: "CreateToolhelp32Snapshot function (tlhelp32.h) Microsoft Learn"

B. Getting Permission: OpenProcess
The Role: Once you have the target's PID, you need a "Handle"—a special security pass that grants your program the right to interact with the target.

How it works in the background: This is a heavy security function. When called, the OS Kernel checks the security descriptor of the target process against the access token of the user running your C++ program. If you ask for PROCESS_VM_READ (permission to read memory) or PROCESS_VM_WRITE (permission to alter memory), the kernel grants a Handle. If you lack administrative rights, the OS blocks you here.

Documentation Search Term: "OpenProcess function (processthreadsapi.h) Microsoft Learn"

C. The Core Operations: ReadProcessMemory & WriteProcessMemory
The Role: These are the functions that actually execute the logic of your Watchdog (reading state) and your Hack (tampering with state).

How it works in the background: When you execute WriteProcessMemory to instantly trigger a hidden escape event, your C++ program triggers a System Call (Syscall). This forces a transition from User-Mode to Kernel-Mode. The Windows Kernel temporarily pauses the target application, physically copies the bytes from your hack's buffer into the target's virtual memory address, verifies the data transfer, and then resumes the target process.

Documentation Search Term: "WriteProcessMemory function (memoryapi.h) Microsoft Learn"
